1. ¿Qué es UML?

  • UML = Unified Modeling Language (Lenguaje Unificado de Modelado)
  • Es un LENGUAJE, NO un proceso/metodología
  • Permite modelar, construir y documentar elementos de un sistema orientado a objetos
  • Se ha convertido en el estándar de facto de la industria

Origen de UML:

  • OMT - Object Modeling Technique (Rumbaugh)
  • Método-Booch (G. Booch)
  • OOSE - Object-Oriented Software Engineering (Jacobson)

Historia:

  • 1997: OMG adopta UML 1.1
  • 2005: UML 2.0 (añade diagrama de estructura compuesta y timing)
  • 2017: UML 2.5
Creadores: Grady Booch, Jim Rumbaugh e Ivar Jacobson

2. Modelado

Modelar = Crear una representación de la realidad

El modelado es el análisis y diseño de aplicaciones software antes de escribir código.

Pros del modelado:

  • Facilita la comprensión para nuevos integrantes
  • Facilita la subcontratación de proyectos
  • Hace más fáciles los cambios
  • Necesario en ciertos entornos (NASA)

Contras del modelado:

  • Modelo completo y realidad ocupan lo mismo
  • La documentación puede ser el código
  • Nadie puede prever de antemano el resultado final

4. Diagrama de Clases

Partes de una clase:

  • Nombre de la clase
  • Atributos
  • Operaciones (métodos)

Tipos de Clase:

TipoSímbolo/Notación
Clase normalCaja rectangular
Clase abstractaNombre en cursiva
Interfaz<<interfaz>>
Enumeración<<enumeration>>
Utility<<utility>>

Visibilidad de atributos y métodos:

SímboloSignificado
+Pública (public)
-Privada (private)
#Protegida (protected)
~Package (sin modificador)

Sintaxis de Atributo:

[visibilidad] nombre [: tipo] ['['multiplicidad']'] [= valor_inicial]

Sintaxis de Operación:

[visibilidad] nombre ['('lista_parametros')'] [: tipo_retorno]

Notas importantes:

  • Atributo static → se subraya
  • Atributo multivaluado (colección) → corchetes []
  • Restricción {ordered} → colección ordenada
  • Clase abstracta → aparece en cursiva

5. Relaciones entre Clases

RelaciónDescripciónSímbolo
Asociación Relación estructural entre objetos Línea continua
Agregación "Tiene un" - ciclo de vida independiente Rombo vacío ◇
Composición "Parte de" - ciclo de vida dependiente (fuerte) Rombo relleno ◆
Herencia/Generalización Hijo hereda propiedades del padre Flecha blanca △
Dependencia Cambio en especificación afecta a otro, pero no al revés Flecha discontinua - - →
Realización Implementa una interfaz Flecha discontinua con triángulo blanco
Diferencia Asociación vs Dependencia:
Asociación = relación estructural permanente
Dependencia = relación temporal/puntual

Relaciones N a N:

Se resuelve con una clase de asociación

6. Programación Orientada a Objetos (POO)

Definición:

Organización del software como una colección de objetos que contienen tanto estructura como comportamiento.

Objetivo principal: REUTILIZACIÓN

Alarmas de que falta OO:

  • "Se nos olvidó hacerlo para X"
  • "Hay que acordarse de probarlo para..."
  • Spaguetti code (if anidados)

Conceptos clave:

  • Clase: Descripción de un conjunto de objetos
  • Objeto: Instancia de una clase
  • Método: Comportamiento/operación
  • Atributo: Característica/propiedad

Modificadores de acceso (Java):

ModificadorAcceso
publicCualquier clase
privateSolo la propia clase
protectedPropia clase + subclases + mismo paquete
(sin modificador)Solo mismo paquete

7. Clases Abstractas e Interfaces

Clase Abstracta:

  • Contiene al menos un método abstracto (sin implementación)
  • Se espera que las subclases lo implementen
  • NO se puede instanciar con new
  • Se representa con el nombre en cursiva

Interfaz:

  • Solo contiene definiciones de métodos y constantes
  • Los métodos son abstractos por defecto
  • Representa una especificación
  • En UML: estereotipo <<interface>>
Diferencia clave:
- Clase abstracta: puede tener estado (atributos con implementación)
- Interfaz: solo contrato, sin implementación (Java 8 permite default/static)

8. Herencia en POO

Relación "ES UN" (is-a)

Características:

  • La subclase hereda todos los atributos y métodos del padre
  • Se usa la palabra clave extends
  • Java solo permite herencia simple (una clase padre)
  • La herencia múltiple se logra mediante interfaces

Clase Genérica:

Asegura que los métodos se aplicarán con un tipo declarado en tiempo de ejecución.

9. Diagrama de Casos de Uso

Elementos:

  • Actor: Entidad externa que usa el sistema
  • Caso de uso: Funcionalidad del sistema
  • Límite del sistema: Define qué está dentro/fuera

Relaciones entre casos de uso:

RelaciónDescripción
Include Un caso de uso siempre usa otro
Extend Un caso de uso puede extender a otro (no necesariamente)
Generalization Herencia entre casos de uso

10. Diagrama de Secuencia

Describe el flujo de información en la interacción entre objetos.

Representa la secuencia de llamadas a métodos entre objetos.

Elementos:

  • Actores/Objetos: En la parte superior
  • Líneas de vida: Líneas verticales
  • Mensajes: Flechas horizontales
  • Activación: Barras verticales sobre las líneas de vida

11. Modelado Ágil

Manifiesto Ágil:

  • No dice "no documentar", dice documentación ligera
  • Problemas de ausencia total de documentación:
    • Dificulta la formación
    • Repetición de trabajo
    • Dificulta la subcontratación

Cuándo documentar (Justo A Tiempo - JIT):

  • ANTES: Parte de requisitos, análisis y diseño
  • DESPUÉS: Formación, petición del cliente, subcontratación

Características de buena documentación:

  • Ordenada y accesible (wiki, ticketing)
  • Colaborativa
