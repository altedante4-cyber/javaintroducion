Ejercicios de Programación Java
Nivel Medio-Alto

Esta colección presenta 20 ejercicios diseñados para consolidar conocimientos de programación orientada a objetos en Java. Los ejercicios cubren estructuras de datos (HashSet, HashMap, ArrayList), patrones de diseño, herencia, polimorfismo, clases abstractas, interfaces y conceptos avanzados en contextos variados y creativos.

1. Sistema de Magia Arcana

Contexto: En un reino fantástico, los magos utilizan runas para lanzar hechizos. Cada runa es un objeto místico que tiene un nombre único (como "Ignis" o "Aqua"), un nivel de poder que va desde 1 hasta 100, y un elemento mágico asociado. Los elementos posibles son: FUEGO, AGUA, TIERRA, AIRE, LUZ y OSCURIDAD.
Descripción del problema:
Los magos necesitan un sistema para gestionar sus grimorios (colecciones personales de runas). Imagina que eres el programador que debe crear este sistema. El mago debe poder:

1. Añadir runas al grimorio: Cuando el mago encuentra una nueva runa, debe poder guardarla en su grimorio. Es importante que NO se permitan duplicados - dos runas con el mismo nombre se consideran la misma runa.

2. Buscar runas por elemento: El mago puede necesitar encontrar todas las runas de un elemento específico (por ejemplo, todas las runas de FUEGO para un ataque incendiario). El sistema debe devolver una lista con todas las runas que coincidan.

3. Calcular el poder total: El grimorio debe poder calcular la suma de los niveles de poder de todas las runas que contiene.

4. Encontrar la runa más poderosa: Dentro de un elemento específico, el mago quiere saber cuál es la runa con mayor poder. Por ejemplo, la runa de fuego más poderosa.
Requisitos técnicos (qué debes usar):
- HashSet: Para almacenar las runas sin duplicados y con búsqueda rápida
- ArrayList: Para devolver listas de runas cuando se busca
- Sobreescribir equals(): Para comparar runas por nombre y evitar duplicados
Estructura de clases recomendada:

1. enum Elemento - Define los elementos mágicos
  • Constantes: FUEGO, AGUA, TIERRA, AIRE, LUZ, OSCURIDAD

2. class Runa - Representa una runa mágica
  • Atributos: nombre (String), poder (int 1-100), elemento (Elemento)
  • Constructor: recibe los tres parámetros
  • Getters: getNombre(), getPoder(), getElemento()
  • Sobreescribir equals() para comparar por nombre

3. class Grimorio - Colección de runas del mago
  • Atributo: runes (HashSet<Runa>)
  • Método: añadirRuna(Runa r) - añade si no existe
  • Método: buscarPorElemento(Elemento e) - devuelve ArrayList<Runa>
  • Método: getPoderTotal() - suma de todos los poderes
  • Método: getRunaMasPoderosa(Elemento e) - la más fuerte de ese elemento

4. class Main - Clase de prueba con menú interactivo
Nota: No es necesario sobrescribir hashCode() en este ejercicio. El comportamiento por defecto del HashSet funciona correctamente si se sobrescribe equals().

2. Simulación de Ecosistema Marino

Contexto: Un biólogo marino necesita simular la vida marina para estudiar las cadenas alimentarias. En este ecosistema existen tres tipos principales de entidades:
- Plancton: Organismos pequeños que flotan y se reproducen, son la base de la cadena alimentaria
- Pez: Animales que nadan, huyen de depredadores y comen plancton para sobrevivir
- Tiburón: Depredadores superiores que cazan peces para alimentarse
Descripción del problema:
Crea un sistema de simulación por turnos donde:

1. Entidades con propiedades: Cada ser vivo tiene:
  • Posición (x, y) en el espacio 2D del océano
  • Energía (cuando llega a 0, la entidad muere)
  • Velocidad de movimiento

2. Comportamiento de los tiburones:
  • Buscan activamente al pez más cercano dentro de su "rango de visión"
  • Se mueven hacia él para cazarlo

3. Comportamiento de los peces:
  • Detectan tiburones cercanos dentro de su "rango de detección"
  • Huyen en dirección opuesta si detectan un depredador

4. Alimentación y muerte:
  • Cuando un tiburón come un pez, gana energía
  • Cuando un pez come plancton, ganha energía
  • Toda entidad pierde energía con el tiempo
  • Cuando energía = 0, la entidad muere y se elimina

5. Simulación por turnos: En cada turno, todas las entidades actúan en orden. El sistema debe mostrar el estado actual (quién sigue vivo, posiciones, energía).
Requisitos técnicos:
- Herencia: Clase padre Entidad, clases hijo Tiburon, Pez, Plancton
- Polimorfismo: Métodos comunes con comportamiento diferente
- ArrayList: Para almacenar listas de entidades
- Arrays bidimensionales: Para representar el mapa del océano
Estructura de clases:

1. abstract class Entidad - Clase base para todas las criaturas
  • Atributos: x, y (double), energia (int), velocidad (double), vivo (boolean)
  • Métodos: mover(double dx, double dy), comer(int energia), distancia(Entidad otra)

2. class Tiburon extends Entidad
  • Atributo adicional: rangoVision (int)
  • Método: buscarPez(ArrayList<Pez>) - encuentra el más cercano

3. class Pez extends Entidad
  • Atributo adicional: rangoDeteccion (int)
  • Método: buscarTiburon(ArrayList<Tiburon>) - detecta peligros

4. class Plancton extends Entidad
  • Método: reproduce() - crea nuevo plancton

5. class Ecosistema - Gestor de la simulación
  • Atributos: tiburones, peces, plancton (ArrayLists), mapa[20][20]
  • Método: simularTurno() - ejecuta un paso de simulación

6. class Main - Programa principal con menú

3. Gestor de Inventario de Videojuego RPG

Contexto: Estás desarrollando un sistema de inventario para un RPG. El jugador tiene un inventario con diferentes tipos de items que debe gestionar. Cada item tiene un valor (en oro), un peso, y una rareza que determina su calidad: COMUN (común), RARO, EPICO o LEGENDARIO.
Descripción del problema:
El sistema de inventario debe:

1. Gestión de peso: El inventario tiene una capacidad máxima de peso (por ejemplo, 100 unidades). No se pueden añadir items que excedan este límite.

2. Organización por rareza: Los items se clasifican automáticamente por su rareza usando un HashMap. La clave es la rareza (String) y el valor es una lista de items de esa rareza.

3. Búsqueda por rareza: Encontrar el item de mayor valor dentro de una rareza específica.

4. Sistema de equipamiento:
  • Solo se puede equipar un arma a la vez
  • Solo se puede equipar una armadura a la vez
  • Equipar mueve el item del inventario a "equipado"
  • Desequipar lo devuelve al inventario

5. Cálculo de estadísticas:
  • Daño total = daño base del personaje + daño del arma equipada
  • Defensa total = defensa de la armadura equipada
Requisitos técnicos:
- HashMap: Para organizar items por rareza
- ArrayList: Para almacenar listas de items
- Herencia: Item como clase padre, subclases para tipos específicos
- Polimorfismo: Métodos comunes en diferentes tipos de items
Estructura de clases:

1. enum Rarity - COMUN, RARO, EPICO, LEGENDARIO

2. enum TipoItem - ARMA, ARMADURA, POCION, MATERIAL

3. abstract class Item - Clase base
  • Atributos: nombre, valor, peso, rarity, tipo
  • Método abstracto: getDescripcion()

4. class Arma extends Item
  • Atributo: daño (int), equipada (boolean)

5. class Armadura extends Item
  • Atributo: defensa (int), equipada (boolean)

6. class Pocion extends Item
  • Atributos: efecto, cantidad

7. class Material extends Item
  • Atributo: cantidad

8. class Inventario - Gestor del inventario
  • Atributos: items (HashMap), capacidadMax, armaEquipada, armaduraEquipada
  • Métodos: añadirItem, eliminarItem, equiparArma, equiparArmadura, getPesoActual, getDañoTotal, getDefensaTotal

4. Protocolo de Comunicación Interestelar

Contexto: Una nave espacial recibe constantemente mensajes codificados de diferentes civilizaciones alienígenas. Cada mensaje tiene un ID único del remitente, el contenido del mensaje, y un nivel de prioridad (1 = muy baja, 5 = muy alta/crítica).
Descripción del problema:
El sistema de comunicaciones debe:

1. Almacenar mensajes: Usar un HashMap donde la clave es el ID del remitente y el valor es el mensaje.

2. Recibir nuevos mensajes: Si un remitente ya existe, el nuevo mensaje sobrescribe el anterior.

3. Buscar mensajes importantes: Recuperar todos los mensajes con prioridad >= 3, ordenados de mayor a menor prioridad.

4. Limpieza automática: Eliminar mensajes de remitentes que no han enviado nada en los últimos X turnos (simulado).

5. Generar reportes: El sistema debe poder generar un reporte con:
  • Total de mensajes almacenados
  • Promedio de prioridad de todos los mensajes
  • Remitente con más mensajes enviados
Requisitos técnicos:
- HashMap: Almacenamiento principal de mensajes por remitente
- ArrayList: Para listas ordenables
- Comparable/Comparator: Para ordenar mensajes por prioridad
- Colecciones: Utilizar Collections.sort() y otros métodos útiles
Estructura de clases:

1. class Mensaje
  • Atributos: id, contenido, prioridad (1-5), remitente, turnoRecibido
  • Implementa Comparable<Mensaje> para ordenar por prioridad

2. class Comunicador
  • Atributos: mensajes (HashMap), ultimoTurnoActivo, contadorTurnos
  • Métodos: recibirMensaje, getMensajesPrioridadAlta, limpiarInactivos, generarReporte

3. class Main

5. Motor de Física de Partículas

Contexto: Un científico necesita simular partículas cargadas en un espacio 2D. Las partículas interactúan entre sí mediante fuerzas eléctricas (atracción si tienen cargas opuestas, repulsión si son del mismo signo). Este es un simulador educativo de física.
Descripción del problema:
El simulador debe:

1. Almacenar partículas: Todas las partículas en un ArrayList.

2. Calcular fuerzas: En cada paso de simulación, calcular la fuerza entre TODAS las parejas de partículas usando la ley de Coulomb (atracción/repulsión según cargas).

3. Actualizar movimiento: Aplicar la segunda ley de Newton (F = m*a) para calcular aceleraciones, y actualizar velocidades y posiciones.

4. Congelar partículas: Una partícula "congelada" no se mueve pero sigue interactuando (ejerciendo fuerzas sobre otras).

5. Consultas del sistema:
  • Partícula más rápida
  • Partícula con mayor aceleración
  • Energía cinética total del sistema (½mv²)
Requisitos técnicos:
- Clases abstractas: Particula como clase padre
- Herencia: Electron, Proton, Neutron, Fotón como subclases
- ArrayList: Almacenamiento de partículas
- Sobrecarga de métodos: Diferentes constructores o métodos con mismos nombres
Estructura de clases:

1. abstract class Particula
  • Atributos: id, x, y, vx, vy, masa, carga, congelada, ax, ay
  • Métodos abstractos: cada subclase define su masa y carga
  • Métodos concretos: calcularAceleracion, mover, congelar, getVelocidad, getEnergiaCinetica

2. class Electrón extends Particula - carga: -1.6e-19, masa: 9.11e-31

3. class Protón extends Particula - carga: 1.6e-19, masa: 1.67e-27

4. class Neutrón extends Particula - carga: 0, masa: 1.67e-27

5. class Fotón extends Particula - masa: 0, carga: 0 (siempre a velocidad c)

6. class SimuladorFísica - Gestor de la simulación
  • Atributos: partículas, constantes físicas
  • Métodos: simularPaso, getParticulaMasRapida, getEnergiaTotal

7. class Main

7. Analizador Léxico de Código

Contexto: Estás construyendo un compilador simplificado. El primer paso es el análisis léxico: convertir el código fuente (texto) en una lista de "tokens" (unidades significativas como palabras reservadas, identificadores, números, operadores, etc.).
Descripción del problema:
El analizador léxico debe:

1. Recibir código fuente: Una cadena con el código a analizar.

2. Identificar tokens: Reconocer y clasificar cada token:
  • PALABRA_RESERVADA: if, else, while, for, return, class, public, void, int...
  • IDENTIFICADOR: nombres de variables, funciones, clases
  • NÚMERO: valores numéricos
  • OPERADOR: +, -, *, /, =, ==, !=, <, >, &&, ||
  • DELIMITADOR: ( ) { } [ ] ; ,
  • CADENA: textos entre comillas

3. Almacenar tokens: Guardar todos los tokens en un ArrayList.

4. Contar frecuencias: Usar HashMap para contar cuántos tokens de cada tipo.

5. Validar balanceo: Verificar que paréntesis, llaves y corchetes estén equilibrados.

6. Reportar errores: Generar lista de errores si encuentra tokens inválidos.
Requisitos técnicos:
- Clases abstractas: Token como clase padre
- Herencia: subclases para cada tipo de token
- HashMap: Contar frecuencia de tipos
- ArrayList: Almacenar lista de tokens
Estructura de clases:

1. enum TipoToken - PALABRA_RESERVADA, IDENTIFICADOR, NUMERO, OPERADOR, DELIMITADOR, CADENA, COMENTARIO, DESCONOCIDO

2. abstract class Token
  • Atributos: tipo, valor, posicion
  • Método abstracto: getTipo()

3. class TokenReservado extends Token - palabras clave de Java

4. class TokenIdentificador extends Token

5. class TokenNumero extends Token - incluye valorNumerico

6. class TokenOperador extends Token

7. class TokenDelimitador extends Token

8. class TokenCadena extends Token

9. class AnalizadorLexico - clase principal
  • Atributos: codigo, tokens, frecuencia, errores
  • Métodos: tokenizar(), validarBalanceo(), getFrecuencias()

8. Simulador de Tráfico Vehicular

Contexto: Una ciudad tiene calles que conectan diferentes puntos. Los vehículos circulan por estas calles siguiendo una ruta desde su origen hasta su destino. Las calles tienen capacidad limitada.
Descripción del problema:
El simulador debe:

1. Gestionar calles: Cada calle tiene:
  • Capacidad máxima de vehículos
  • Número actual de vehículos (flujo)
  • Longitud

2. Vehículos con rutas: Cada vehículo tiene:
  • Origen y destino
  • Ruta: lista de calles a seguir
  • Velocidad

3. Lógica de tráfico:
  • Si una calle está llena, el vehículo debe esperar
  • Cada segundo, los vehículos avanzan una "unidad" de calle
  • Al llegar al final de una calle, pasan a la siguiente

4. Estado del sistema: Mostrar:
  • Vehículos en tránsito
  • Vehículos que llegaron a destino
  • Vehículos esperando
Requisitos técnicos:
- Herencia: Vehiculo clase padre, Coche, Camion, Moto como hijos
- HashMap: Mapa de calles por nombre
- ArrayList: Listas de vehículos
- Polimorfismo: Diferentes velocidades y capacidades
Estructura de clases:

1. class Calle
  • Atributos: nombre, capacidad, vehiculosActuales, longitud
  • Métodos: añadirVehiculo, removeVehiculo, estaLlena

2. abstract class Vehiculo
  • Atributos: id, origen, destino, ruta, posicionActual, velocidad, esperando, llegada
  • Método abstracto: getTipo()

3. class Coche extends Vehiculo - velocidad: 3

4. class Camión extends Vehiculo - velocidad: 1, capacidad: 2

5. class Moto extends Vehiculo - velocidad: 4

6. class Ciudad - gestror del tráfico
  • Atributos: calles (HashMap), vehiculos, vehiculosEsperando, vehiculosLlegados
  • Método: simularSegundo(), getEstado()

9. Sistema de Mensajería Cifrada

Contexto: Una aplicación de mensajería segura usa diferentes algoritmos de cifrado según el nivel de seguridad requerido por el usuario. Los métodos van desde simples (César) hasta más complejos.
Descripción del problema:
El sistema debe:

1. Gestionar mensajes: Cada mensaje tiene contenido, destinatario y nivel de seguridad.

2. Almacenar por destinatario: Usar HashMap donde la clave es el nombre del destinatario.

3. Aplicar cifrado según nivel:
  • Nivel 1 (César): Desplazar cada letra X posiciones (X = nivel del usuario)
  • Nivel 2 (Vigenère): Usar el nombre del destinatario como clave
  • Nivel 3 (Híbrido): Invertir la cadena + cifrado César

4. Descifrado: Revertir el proceso usando el mismo nivel de seguridad.

5. Buscar mensajes: Recuperar todos los mensajes de un destinatario específico.
Requisitos técnicos:
- Interfaces: Cifrador con métodos cifrar() y descifrar()
- Implementaciones: Clases que implementan Cifrador para cada algoritmo
- HashMap: Almacenar mensajes por destinatario
Estructura de clases:

1. class Mensaje
  • Atributos: id, contenido, destinatario, nivelSeguridad, cifrado

2. interface Cifrador
  • Métodos: cifrar(String texto, String clave): String
  • Método: descifrar(String texto, String clave): String

3. class CifradorCésar implements Cifrador
  • Atributo: desplazamiento

4. class CifradorVigenère implements Cifrador
  • Atributo: clave

5. class CifradorHíbrido implements Cifrador - invertir + César

6. class GestorMensajes
  • Atributos: mensajes (HashMap), cifradores (HashMap)
  • Métodos: enviarMensaje, recibirMensaje, cifrar, descifrar

7. class Main
